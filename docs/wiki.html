Web Scraper Concorrente em Go

Um comparador de preços que consulta várias lojas ao mesmo tempo, construído para demonstrar o modelo de concorrência da linguagem Go — goroutines, channels e select — e compará-lo com C e Python.

Linguagem: Go 1.21+ Módulos: 3 Integrantes: 5 Comparações: C · Python

Visão geral

Este projeto resolve um problema simples do dia a dia — descobrir em qual loja um produto está mais barato — e o usa como pretexto para exercitar aquilo que a linguagem Go faz de melhor: concorrência de forma nativa, legível e segura.

A ideia central é que, ao pesquisar preços em dez lojas, o computador passa a maior parte do tempo esperando a resposta de cada site, e não calculando. Se as dez consultas forem disparadas simultaneamente, em vez de uma após a outra, o tempo total cai de "a soma de todas as esperas" para "a espera da loja mais lenta". É exatamente esse ganho que o projeto mede e demonstra.

Para cobrir bem o tema, o trabalho foi dividido em três módulos independentes:

  • webscraper (mock) — sobe um servidor HTTP local que simula 10 lojas com latências diferentes. É uma demonstração determinística, que roda sem depender da internet e sempre exibe o comportamento de timeout.
  • webscraper-real — consulta uma API pública de verdade (dummyjson.com), com dados reais de produtos, preços e avaliações.
  • python/comparacao.py — resolve o mesmo problema em Python (sequencial, com threads e com pool) para servir de referência de desempenho contra o Go.

01A linguagem Go

Histórico

Go (ou golang) nasceu dentro do Google em 2007, das mãos de Robert Griesemer, Rob Pike e Ken Thompson — os dois últimos com raízes profundas no Unix e na própria linguagem C. A motivação era concreta e dolorosa: na escala do Google, as compilações de C++ eram lentíssimas, a gestão de dependências era confusa e escrever software concorrente era difícil e propenso a erros.

A resposta foi projetar uma linguagem deliberadamente minimalista: rápida de compilar, fácil de ler e com concorrência embutida na própria sintaxe. Go foi anunciada como projeto open source em novembro de 2009, sob licença BSD. A versão 1.0 saiu em março de 2012, acompanhada da célebre "promessa de compatibilidade do Go 1": código escrito naquela época ainda compila nas versões atuais.

Versão e características

A versão estável atual é a série Go 1.24 (2025). Em resumo, Go é:

  • Compilada para código de máquina nativo (não interpretada), gerando um binário único.
  • Estaticamente tipada, com verificação de tipos em tempo de compilação.
  • Gerenciada por garbage collector — o programador não libera memória manualmente.
  • Da família sintática do C, mas com apenas 25 palavras-chave, o que mantém a linguagem pequena e memorizável.
Onde Go é usado Boa parte da infraestrutura moderna é escrita em Go: Docker, Kubernetes, Terraform, Prometheus e Grafana. Isso reforça o domínio natural da linguagem — serviços de rede concorrentes, exatamente o caso deste projeto.

02O projeto: premissas, usuário e domínio

Domínio

O domínio é web scraping concorrente aplicado a comparação de preços. O programa coleta, via requisições HTTP, o preço de um mesmo produto em várias lojas, decodifica a resposta em JSON e monta um ranking do mais barato ao mais caro.

Usuário

Há dois perfis de usuário. O primeiro é o consumidor que quer o menor preço sem precisar abrir dez abas no navegador. O segundo — e principal, dado o contexto acadêmico — é o estudante que quer ver a concorrência acontecendo: o terminal mostra, em tempo real, cada loja respondendo na ordem da sua velocidade, e as lojas lentas sendo descartadas.

Premissa central

A premissa que justifica toda a arquitetura é: o gargalo é a rede, não o processador. Como cada requisição passa quase todo o tempo esperando a resposta do servidor, disparar as requisições em paralelo não sobrecarrega a CPU — apenas aproveita o tempo ocioso de espera. Por isso o tempo total tende ao tempo da requisição mais lenta, e não à soma de todas.

Uma premissa secundária, de engenharia, é a de robustez: uma loja que demora demais não pode travar o programa inteiro. Daí a necessidade de um timeout por loja, implementado com select.

04Construtores

O grande diferencial de Go é que a concorrência não vem de uma biblioteca — ela é feita de palavras-chave da própria linguagem. Três construtores sustentam o projeto inteiro.

Goroutines — go

Uma goroutine é uma função executando concorrentemente. Basta prefixar a chamada com a palavra go. Diferente de uma thread do sistema operacional, uma goroutine começa com apenas ~2 KB de pilha e é escalonada pelo próprio runtime de Go sobre os núcleos disponíveis — por isso é viável criar milhares delas. No projeto, dispara-se uma goroutine por loja.

Channels — chan

Um channel é um "cano" tipado por onde as goroutines trocam dados com segurança, sem que o programador precise de locks manuais. A filosofia de Go é: "não se comunique compartilhando memória; compartilhe memória comunicando-se". O operador <-ch recebe um valor e bloqueia até que haja um, o que sincroniza as goroutines naturalmente.

Select — select

O select espera vários canais ao mesmo tempo e reage ao primeiro que estiver pronto. É o coração do timeout do projeto: uma goroutine interna faz o request HTTP e envia o resultado por um canal; em paralelo, time.After devolve um valor após o prazo. O select escolhe quem chegar primeiro — resposta válida ou estouro de tempo.

Demais construtores usados

  • struct — para modelar Result, Product e a configuração das lojas.
  • defer — garante resp.Body.Close() ao final da função, mesmo com erro.
  • slice e map — coleções nativas para acumular e ordenar resultados.
  • canal com buffer 1 — make(chan Result, 1) evita que a goroutine interna fique presa (vazamento) caso o timeout vença primeiro.

05Legibilidade

Legibilidade é o quanto um código é fácil de ler e entender. Go trata isso como prioridade de projeto:

  • gofmt — um formatador oficial embutido. Não existe debate sobre estilo: há uma forma canônica, e a ferramenta a aplica. Todo código Go se parece com todo código Go.
  • 25 palavras-chave — a linguagem inteira cabe na cabeça; há pouca sintaxe para decorar.
  • Erros explícitos — o padrão if err != nil deixa o caminho de falha visível na leitura, sem exceções "saltando" de lugares distantes.
  • Sem pré-processador, headers ou macros — o que se lê é o que executa.

Comparação com C

A diferença fica evidente na mesma tarefa — buscar uma URL e ler o JSON de resposta. Em Go, quatro linhas com a biblioteca padrão; em C, é preciso a libcurl (dependência externa) só para o HTTP, e ainda um parser JSON de terceiros à parte:

Go — biblioteca padrão
resp, err := http.Get(url)
if err != nil { return err }
defer resp.Body.Close()
json.NewDecoder(resp.Body).
     Decode(&data)
C — libcurl + parser externo
#include <curl/curl.h>
CURL *c = curl_easy_init();
curl_easy_setopt(c, CURLOPT_URL, url);
curl_easy_setopt(c, CURLOPT_WRITEFUNCTION, cb);
if (curl_easy_perform(c) != CURLE_OK) {...}
curl_easy_cleanup(c);
/* + cJSON à parte p/ ler o JSON */

Somam-se a isso, em C, os #include/#define, a aritmética de ponteiros e a ausência de um formatador único — cada projeto adota seu próprio estilo, o que dificulta a leitura por terceiros.

Comprovado no projeto A partir do Linux, o mesmo código-fonte gerou um executável Windows com um único comando: GOOS=windows go build. Essa portabilidade trivial é uma das maiores vantagens de escrita de Go.

07Confiabilidade

Confiabilidade é a capacidade da linguagem de evitar que o programa se comporte de forma incorreta. Go elimina, por construção, classes inteiras de bugs que assombram o C:

RiscoGoC
Memóriagarbage collector automáticomalloc/free manual
Ponteiro soltonão há use-after-freedangling pointer, double-free
Bufferchecagem de limites em runtimeestouro de buffer silencioso
Corrida de dadosgo run -race detectasem detector nativo
Falhapanic explícitocomportamento indefinido (UB)

No projeto, a confiabilidade aparece de forma concreta no canal com buffer 1: se o timeout do select vencer antes da resposta, a goroutine interna ainda consegue enviar seu resultado sem ficar bloqueada para sempre — evitando um goroutine leak. O código documenta exatamente isso.

Bug real encontrado e corrigido A versão original chamava syscall.NewLazyDLL (API exclusiva do Windows) sem build tags. O código não compilava em Linux/macOS, pois o compilador exige o símbolo mesmo com o teste runtime.GOOS. A correção idiomática foi separar a função em dois arquivos com build tags — ansi_windows.go (Windows) e ansi_other.go (no-op). Isso restaurou a compilação em todas as plataformas.

09Exemplos de código

O fluxo do módulo mock cabe em três ideias: disparar, esperar e coletar.

webscraper/main.go
// 1. canal com buffer: goroutines enviam sem travar umas às outras
ch := make(chan Result, len(mock.Stores))

// 2. dispara TODAS as lojas ao mesmo tempo — uma goroutine cada
for _, store := range mock.Stores {
    url := fmt.Sprintf("http://localhost:%d%s", serverPort, store.Path)
    go scrapePrice(store.Name, url, ch)
}

// 3. coleta pelo canal — a ordem é a VELOCIDADE da loja, não a da lista
for range mock.Stores {
    r := <-ch          // bloqueia até QUALQUER goroutine enviar
    results = append(results, r)
    printResult(r)
}

E o coração do timeout, na função scrapePrice, com o select disputando entre a resposta HTTP e o relógio:

o padrão SELECT
done := make(chan Result, 1)   // buffer 1 evita goroutine leak

go func() { /* faz o http.Get e envia em 'done' */ }()

select {
case r := <-done:              // resposta chegou a tempo ✓
    ch <- r
case <-time.After(storeTimeout): // estourou o prazo ⚠
    ch <- Result{Store: name, TimedOut: true}
}

11Comparação com C

Go e C compartilham a mesma família de sintaxe, mas separa-os décadas de evolução em segurança e produtividade. A tabela consolida os pontos discutidos ao longo desta wiki:

CritérioGoC
Concorrêncianativa: go, channels, selectmanual: pthreads + mutex/locks
Memóriagarbage collectormalloc/free manual
Segurançasem ponteiro inseguro; race detectorbuffer overflow, UB, segfault
HTTP + JSONbibliotecas padrão embutidasdependências externas (libcurl, cJSON)
Compilaçãorápida; binário estático únicorápida, mas link/deps podem doer
Cross-compileGOOS=windows go buildtoolchain cruzado dedicado
Legibilidadegofmt único; 25 palavras-chavemacros, headers, estilos divergentes

Em resumo: C continua imbatível em sistemas embarcados e tempo-real duro, onde o controle absoluto de memória e a previsibilidade importam mais que a produtividade. Já para serviços de rede concorrentes — o domínio deste projeto — Go entrega desempenho equivalente com muito menos risco e muito menos código.

12Comparação Python × Go

O módulo em Python resolve o mesmo problema de três formas, para servir de referência. Os tempos foram medidos na mesma execução:

ImplementaçãoTempovs sequencialObservação
Python sequencial6.01s1.0×sem concorrência
Python threads + queue7.29s0.8×GIL só libera em I/O
Python ThreadPoolExecutor3.33s1.8×idiomático, mas ainda limitado pelo GIL
Go goroutines0.29s20.7×paralelismo real, 2KB/goroutine

A diferença brutal vem do GIL (Global Interpreter Lock) do Python, que permite apenas uma thread executando bytecode Python por vez. Em tarefas de I/O o GIL é liberado durante a espera, então as threads ajudam um pouco — mas continuam sendo threads pesadas do sistema operacional. As goroutines de Go, ao contrário, rodam em paralelismo real sobre vários núcleos e custam uma fração da memória. Daí o abismo de desempenho.

14Como executar

Com o Go instalado (1.21 ou superior), a partir da raiz do repositório:

bash
# módulo 1 — demo com lojas simuladas (não precisa de internet)
cd webscraper && go run .

# módulo 2 — API real (requer conexão com a internet)
cd webscraper-real && go run .

# módulo 3 — comparação em Python
cd python && python3 comparacao.py

# compilar um binário (ex.: para Windows a partir do Linux)
GOOS=windows go build -o webscraper.exe .
