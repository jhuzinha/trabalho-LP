Comparador de Preços Concorrente em Go

Uma aplicação que consulta várias lojas ao mesmo tempo para achar a melhor oferta, construída para demonstrar o modelo de concorrência de Go — goroutines, channels, worker pool e context — com uma versão em Python (ThreadPoolExecutor) como comparação didática.

Linguagem: Go 1.21+ Módulo: trabalho-lp Integrantes: 5 Comparações: C · Python

Visão geral

Este projeto resolve um problema simples do dia a dia — descobrir em qual loja um produto está mais barato — e o usa como pretexto para exercitar aquilo que a linguagem Go faz de melhor: concorrência de forma nativa, legível e segura.

A ideia central é que, ao pesquisar preços em dez lojas, o computador passa a maior parte do tempo esperando a resposta de cada site, e não calculando. Se as dez consultas forem disparadas simultaneamente, em vez de uma após a outra, o tempo total cai de "a soma de todas as esperas" para "a espera da loja mais lenta". É exatamente esse ganho que o projeto mede e demonstra.

Para tornar a demonstração reprodutível e a comparação entre linguagens justa, o trabalho tem três peças que conversam entre si:

  • Servidor mock (internal/mockserver) — sobe um servidor HTTP local que simula 10 lojas com latências fixas. Como os atrasos são fixos, Go e Python enfrentam exatamente o mesmo cenário, sem depender da internet.
  • Scraper em Go (cmd/scraper + internal/scraper) — a implementação principal. Consulta as lojas com um worker pool de goroutines e compara a execução concorrente com uma baseline sequencial.
  • Coletor em Python (python/comparador_python.py) — resolve o mesmo problema com ThreadPoolExecutor, servindo de referência para discutir quando cada linguagem vale mais a pena.

01A linguagem Go

Histórico

Go (ou golang) nasceu dentro do Google em 2007, das mãos de Robert Griesemer, Rob Pike e Ken Thompson — os dois últimos com raízes profundas no Unix e na própria linguagem C. A motivação era concreta e dolorosa: na escala do Google, as compilações de C++ eram lentíssimas, a gestão de dependências era confusa e escrever software concorrente era difícil e propenso a erros.

A resposta foi projetar uma linguagem deliberadamente minimalista: rápida de compilar, fácil de ler e com concorrência embutida na própria sintaxe. Go foi anunciada como projeto open source em novembro de 2009, sob licença BSD. A versão 1.0 saiu em março de 2012, acompanhada da célebre "promessa de compatibilidade do Go 1": código escrito naquela época ainda compila nas versões atuais.

Versão e características

A série estável atual é a Go 1.24 (2025); este projeto declara go 1.21 no go.mod. Em resumo, Go é:

  • Compilada para código de máquina nativo (não interpretada), gerando um binário único.
  • Estaticamente tipada, com verificação de tipos em tempo de compilação.
  • Gerenciada por garbage collector — o programador não libera memória manualmente.
  • Da família sintática do C, mas com apenas 25 palavras-chave, o que mantém a linguagem pequena e memorizável.
Onde Go é usado Boa parte da infraestrutura moderna é escrita em Go: Docker, Kubernetes, Terraform, Prometheus e Grafana. Isso reforça o domínio natural da linguagem — serviços de rede concorrentes, exatamente o caso deste projeto.

02O projeto: premissas, usuário e domínio

Domínio

O domínio é comparação de preços em e-commerce. O programa coleta, via requisições HTTP, o preço de um mesmo produto em várias lojas, decodifica a resposta em JSON e monta um ranking do mais barato ao mais caro.

Usuário

Há dois perfis de usuário. O primeiro é o consumidor que quer o menor preço sem precisar abrir dez abas no navegador. O segundo — e principal, dado o contexto acadêmico — é o estudante que quer ver a concorrência acontecendo: o terminal mostra, em tempo real, cada loja respondendo na ordem da sua velocidade, e as lojas lentas sendo descartadas por timeout.

Premissa central

A premissa que justifica toda a arquitetura é: o gargalo é a rede, não o processador. Como cada requisição passa quase todo o tempo esperando a resposta do servidor, disparar as requisições em paralelo não sobrecarrega a CPU — apenas aproveita o tempo ocioso de espera. Por isso o tempo total tende ao tempo da requisição mais lenta, e não à soma de todas.

Uma premissa secundária, de engenharia, é a de robustez: uma loja que demora demais não pode travar o programa inteiro. Daí a necessidade de um timeout por loja, implementado com context.WithTimeout — que cancela a requisição HTTP de verdade quando o prazo estoura.

03Estrutura do código

O repositório é um único módulo Go (trabalho-lp), organizado no layout idiomático cmd/ + internal/: cada pacote tem uma responsabilidade única e clara.

árvore do projeto
trabalho-LP/
├── cmd/
│   ├── scraper/main.go       # app principal: sobe o mock e roda o scraper
│   └── mockserver/main.go    # servidor mock standalone (p/ o Python separado)
├── internal/
│   ├── model/                # structs Store e Result + construtores New...
│   ├── scraper/              # worker pool: RunConcurrent / RunSequential / Fetch
│   ├── mockserver/           # servidor HTTP das 10 lojas simuladas
│   └── report/               # saída colorida no terminal + CSV/JSON
├── python/
│   └── comparador_python.py  # comparação com ThreadPoolExecutor
├── scripts/                  # comparar_go_python.{sh,ps1,bat}
├── go.mod                    # módulo único: trabalho-lp (go 1.21)
├── Makefile
└── reports/                  # CSV/JSON gerados na execução

Diferente de um projeto com vários go.mod, aqui há um módulo só: os pacotes se importam entre si por caminho (trabalho-lp/internal/scraper, etc.). A pasta internal/ não é decorativa — o compilador de Go proíbe que código de fora do módulo importe qualquer coisa dentro dela, o que deixa explícito o que é API pública (cmd/) e o que é detalhe interno.

04Construtores

Go não tem construtor formal como Java ou C++. O padrão idiomático é uma função NewTipo que inicializa a struct e valida os dados, impedindo que o resto do sistema receba um valor inconsistente.

Os construtores do projeto

Quatro funções New... sustentam a montagem do programa, cada uma com sua validação:

  • model.NewStore(name, url) — recusa nome ou URL vazios.
  • scraper.NewConfig(timeout, workers) — exige timeout e número de workers maiores que zero.
  • scraper.NewScraper(config) — cria o cliente HTTP já com a configuração validada.
  • mockserver.NewServer(port, stores) — valida a porta e exige pelo menos uma loja.
internal/scraper/scraper.go
// NewConfig é o construtor de Config: inicializa e valida.
func NewConfig(timeout time.Duration, workers int) (Config, error) {
    if timeout <= 0 {
        return Config{}, errors.New("timeout deve ser maior que zero")
    }
    if workers <= 0 {
        return Config{}, errors.New("quantidade de workers deve ser maior que zero")
    }
    return Config{Timeout: timeout, Workers: workers}, nil
}

Os blocos de concorrência da linguagem

Além dos construtores, o projeto se apoia nas palavras-chave que fazem a concorrência de Go ser parte da própria linguagem, e não de uma biblioteca:

  • go — dispara uma goroutine: uma função rodando concorrentemente, com início de apenas ~2 KB de pilha, escalonada pelo runtime de Go.
  • chan — um channel tipado, o "cano" por onde as goroutines trocam dados com segurança, sem locks manuais. No projeto há dois: jobs e results.
  • select — espera vários canais ao mesmo tempo; usado para respeitar o cancelamento (<-ctx.Done()) enquanto envia trabalho ou resultado.
  • sync.WaitGroup — espera de forma segura todos os workers terminarem antes de fechar o canal de resultados.
  • context.WithTimeout — carrega o prazo por requisição e cancela de verdade a chamada HTTP lenta.

05Legibilidade

Legibilidade é o quanto um código é fácil de ler e entender. Go trata isso como prioridade de projeto:

  • gofmt — um formatador oficial embutido. Não existe debate sobre estilo: há uma forma canônica, e a ferramenta a aplica. Todo código Go se parece com todo código Go.
  • Pacotes com uma responsabilidade cada — model, scraper, mockserver e report separam claramente dados, lógica, servidor e saída.
  • Erros explícitos — o padrão if err != nil deixa o caminho de falha visível na leitura, sem exceções "saltando" de lugares distantes.
  • Sem pré-processador, headers ou macros — o que se lê é o que executa.

Comparação com C

A diferença fica evidente na mesma tarefa — buscar uma URL e ler o JSON de resposta. Em Go, quatro linhas com a biblioteca padrão; em C, é preciso a libcurl (dependência externa) só para o HTTP, e ainda um parser JSON de terceiros à parte:

Go — biblioteca padrão
resp, err := http.Get(url)
if err != nil { return err }
defer resp.Body.Close()
json.NewDecoder(resp.Body).
     Decode(&data)
C — libcurl + parser externo
#include <curl/curl.h>
CURL *c = curl_easy_init();
curl_easy_setopt(c, CURLOPT_URL, url);
curl_easy_setopt(c, CURLOPT_WRITEFUNCTION, cb);
if (curl_easy_perform(c) != CURLE_OK) {...}
curl_easy_cleanup(c);
/* + cJSON à parte p/ ler o JSON */

Somam-se a isso, em C, os #include/#define, a aritmética de ponteiros e a ausência de um formatador único — cada projeto adota seu próprio estilo, o que dificulta a leitura por terceiros.

07Confiabilidade

Confiabilidade é a capacidade da linguagem de evitar que o programa se comporte de forma incorreta. Go elimina, por construção, classes inteiras de bugs que assombram o C:

RiscoGoC
Memóriagarbage collector automáticomalloc/free manual
Ponteiro soltonão há use-after-freedangling pointer, double-free
Bufferchecagem de limites em runtimeestouro de buffer silencioso
Corrida de dadosgo test -race detectasem detector nativo
Falhapanic explícitocomportamento indefinido (UB)

No projeto, a confiabilidade aparece de forma concreta em várias camadas:

  • Validação nos construtores — NewConfig, NewScraper, NewStore e NewServer rejeitam entradas inválidas antes de qualquer requisição.
  • Erros viram resultado, não crash — cada Fetch devolve um Result de sucesso, timeout ou erro; uma loja com falha não derruba a coleta inteira.
  • Validação de status HTTP, JSON e preço — respostas com status diferente de 200, JSON inválido ou preço ≤ 0 são tratadas como erro explícito.
  • Timeout real com context.WithTimeout — a requisição lenta é cancelada de verdade, não apenas ignorada.
Testado neste projeto O worker pool compartilha canais entre várias goroutines — terreno clássico para race conditions. Rodamos go test -race ./... e todos os testes passam sem corrida de dados detectada, o que dá confiança de que a comunicação por channels está correta.

Além dos critérios de linguagem, pesam fatores práticos de adoção. Go sai bem em quase todos:

Custo controlado com worker pool Em vez de disparar uma goroutine por loja sem limite, o projeto usa um worker pool: um número fixo de workers (por padrão 5) consome as lojas de um canal jobs. Assim, mesmo com 10 mil lojas, a concorrência — e o custo de CPU, memória e conexões de rede — permanece previsível e limitado.

09Exemplos de código

O coração do projeto é o worker pool em internal/scraper/scraper.go. A ideia cabe em três papéis: um produtor que enfileira lojas, N workers que consomem, e um coletor que junta os resultados.

RunConcurrent — worker pool
jobs := make(chan model.Store)
results := make(chan model.Result)

// 1. sobe N workers como goroutines
var wg sync.WaitGroup
wg.Add(workerCount)
for id := 1; id <= workerCount; id++ {
    go s.worker(ctx, id, jobs, results, &wg)
}

// 2. produtor: envia as lojas para o canal jobs e o fecha
go func() {
    defer close(jobs)
    for _, store := range stores {
        select {
        case <-ctx.Done(): return
        case jobs <- store:
        }
    }
}()

// 3. quando todos os workers terminam, fecha o canal de results
go func() { wg.Wait(); close(results) }()

// 4. coleta os resultados na ordem em que chegam
for result := range results {
    collected = append(collected, result)
}

Cada worker é uma goroutine simples: pega uma loja do canal jobs, consulta e devolve o Result pelo canal results, respeitando o cancelamento do contexto.

worker + Fetch (timeout com context)
func (s *Scraper) worker(ctx context.Context, id int,
    jobs <-chan model.Store, results chan<- model.Result, wg *sync.WaitGroup) {
    defer wg.Done()
    for store := range jobs {
        results <- s.Fetch(ctx, store)
    }
}

// Fetch cancela a requisição de verdade quando estoura o prazo.
requestCtx, cancel := context.WithTimeout(ctx, s.config.Timeout)
defer cancel()
req, _ := http.NewRequestWithContext(requestCtx, http.MethodGet, store.URL, nil)
resp, err := s.client.Do(req)
if errors.Is(requestCtx.Err(), context.DeadlineExceeded) {
    return model.NewTimeoutResult(store.Name, elapsed, "sem resposta a tempo")
}

Os números abaixo foram medidos em execução real do projeto, com 10 lojas, 5 workers e timeout de 2,5 s. Oito lojas responderam dentro do prazo; duas (AliExpress e Carrefour, propositalmente lentas) foram descartadas pelo timeout do context.

A melhor oferta encontrada foi a Shopee, por R$ 1199,00. Os resultados são salvos em reports/resultados_concorrentes.csv e .json.

Por que 4,0 s e não 2,5 s? Com 5 workers para 10 lojas, cada worker atende duas lojas em sequência. O tempo total é o do worker que pegou o par mais lento — não a soma de todas as esperas (14,9 s), mas também não uma única loja. É o custo, e o controle, do worker pool.

11Comparação com C

Go e C compartilham a mesma família de sintaxe, mas separa-os décadas de evolução em segurança e produtividade. A tabela consolida os pontos discutidos ao longo desta wiki:

Em resumo: C continua imbatível em sistemas embarcados e tempo-real duro, onde o controle absoluto de memória e a previsibilidade importam mais que a produtividade. Já para serviços de rede concorrentes — o domínio deste projeto — Go entrega desempenho equivalente com muito menos risco e muito menos código.

12Comparação Python × Go

O coletor em Python resolve exatamente o mesmo cenário — as mesmas 10 lojas, com os mesmos preços e atrasos — usando ThreadPoolExecutor em vez de goroutines. Os tempos abaixo foram medidos na mesma máquina:

ImplementaçãoConcorrenteSequencialSpeedup
Go — worker pool4.00s14.91s3.7×
Python — ThreadPoolExecutor4.01s14.93s3.72×

A conclusão honesta é que, neste problema, Go e Python empatam. E isso é esperado: como a tarefa é I/O-bound (quase todo o tempo é espera de rede), o GIL do Python é liberado durante cada espera, então as threads conseguem sobrepor as consultas quase tão bem quanto as goroutines. O gargalo é a rede, e a rede é a mesma para os dois.

Go continua trazendo vantagens que não aparecem neste microbenchmark, mas importam em produção:

  • goroutines são muito mais leves que threads do SO (~2 KB vs ~1 MB), então escalam para muito mais tarefas simultâneas;
  • channels e context fazem parte da linguagem — cancelamento e comunicação são idiomáticos, não bibliotecas;
  • o binário estático único é trivial de distribuir e conteinerizar;
  • para tarefas CPU-bound, o GIL serializaria as threads Python, enquanto Go usa paralelismo real em vários núcleos.
A conclusão madura Go não é "sempre mais rápido". Para este caso I/O-bound, Python resolve muito bem. Go se destaca quando concorrência, rede e execução em produção são partes centrais do problema — e é aí que goroutines, channels e worker pool viram uma vantagem estrutural, não só de velocidade.

14Como executar

Com o Go instalado (1.21 ou superior), a partir da raiz do repositório:

bash
# só a versão Go (sobe o mock interno automaticamente)
go run ./cmd/scraper

# comparativo Go vs Python lado a lado
make compare                 # Linux/macOS/Git Bash/WSL
# Windows:  .\scripts\comparar_go_python.bat

# testes e detector de corrida de dados
make test                    # go test ./...
make race                    # go test -race ./...

# só a versão Python (mock interno em Python)
python3 python/comparador_python.py --mock-interno

# compilar um binário Windows a partir do Linux
GOOS=windows go build -o scraper.exe ./cmd/scraper

Parâmetros opcionais em Go: -workers=10, -timeout=1500ms, -porta=18080. No Python, os equivalentes são --workers, --timeout e --porta.
