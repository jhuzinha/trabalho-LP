02 A linguagem

Linguagem: histórico e versão

  • 2007 — concebida no Google por Robert Griesemer, Rob Pike e Ken Thompson (Pike e Thompson vêm do Unix e do C).
  • Nov/2009 — anunciada como projeto open source (licença BSD, gratuita).
  • Mar/2012 — Go 1.0 e a "promessa de compatibilidade": código de 2012 ainda compila hoje.
  • Hoje — série Go 1.24 (2025); este projeto declara go 1.21 no go.mod.

A motivação foi concreta: builds de C++ lentíssimos, dependências emaranhadas e concorrência difícil. A resposta foi uma linguagem minimalista — só 25 palavras-chave.

04 Critério de avaliação

Construtores

Go não tem construtor formal como Java/C++. O idioma é uma função NewTipo que inicializa e valida.

  • model.NewStore — recusa nome ou URL vazios.
  • scraper.NewConfig — exige timeout e workers > 0.
  • scraper.NewScraper — cria o cliente HTTP já validado.
  • mockserver.NewServer — valida porta e lista de lojas.

E a concorrência é feita de palavras-chave da linguagem: go, chan, select, sync.WaitGroup e context.

Go — construtor que valida
func NewConfig(timeout time.Duration, workers int) (Config, error) { if timeout <= 0 { return Config{}, errors.New("timeout > 0") } if workers <= 0 { return Config{}, errors.New("workers > 0") } return Config{timeout, workers}, nil }
C — sem construtor nem validação embutida
/* struct crua: quem chama precisa lembrar de validar */ Config c; c.timeout = t; /* nada impede t <= 0 */ c.workers = w; /* nada impede w <= 0 */ /* validação fica espalhada, feita à mão em cada uso */

Em Go, o construtor NewTipo concentra a validação num só lugar e devolve error — o resto do sistema nunca recebe um valor inconsistente.

05 Critério de avaliação

Legibilidade

  • gofmt — formatador oficial embutido. Não há debate sobre estilo: existe uma forma canônica e a ferramenta aplica.
  • Pacotes com uma responsabilidade — model, scraper, mockserver, report separam dados, lógica, servidor e saída.
  • Erros explícitos — if err != nil deixa o caminho de falha visível na leitura, sem exceções escondidas.
  • Sem pré-processador, sem headers, sem macros — o que você lê é o que executa.

A prova é lado a lado: a mesma busca HTTP + leitura de JSON nas duas linguagens.

Go — net/http + encoding/json (padrão)
resp, err := http.Get(url) if err != nil { return err } defer resp.Body.Close() json.NewDecoder(resp.Body).Decode(&data)
C — libcurl + parser JSON externo
#include <curl/curl.h> /* dependência externa */ CURL *c = curl_easy_init(); curl_easy_setopt(c, CURLOPT_URL, url); curl_easy_setopt(c, CURLOPT_WRITEFUNCTION, cb); if (curl_easy_perform(c) != CURLE_OK) { ... } curl_easy_cleanup(c); /* + cJSON à parte só para LER o JSON */

Além disso, em C a legibilidade sofre com #include/#define, aritmética de ponteiros e estilos divergentes entre projetos — Go tem gofmt, um só jeito.

06 Critério de avaliação

Capacidade de escrita

net/http

Cliente HTTP embutido

Um http.Get(url) e pronto. Em C seria libcurl (dependência externa) + configuração manual.

encoding/json · csv

JSON → struct e relatórios

Decodifica para tipos com tags e grava CSV/JSON na stdlib. Em C não há JSON nativo — precisa de cJSON/jansson.

go · chan · worker pool

Concorrência de 1ª classe

Sem pthread_create, mutex e join manuais como em C.

GOOS=windows go build

Cross-compilation trivial

Comprovado neste teste: o mesmo código gerou um .exe Windows a partir do Linux, com um comando.

bash — feito durante o teste deste projeto
$ go build ./cmd/scraper ✓ build OK $ GOOS=windows go build -o scraper.exe ./cmd/scraper ✓ PE32+ Windows OK
07 Critério de avaliação

Confiabilidade

  • Garbage collector — sem malloc/free. Adeus use-after-free, double-free e vazamentos manuais.
  • Validação nos construtores — NewConfig, NewStore e NewServer rejeitam entrada inválida antes de qualquer requisição.
  • Erro vira resultado, não crash — status HTTP, JSON e preço são validados; uma loja com falha não derruba a coleta.
  • Timeout real — context.WithTimeout cancela de verdade a requisição de uma loja lenta.
  • Race detector — go test -race rodou limpo sobre o worker pool neste projeto.

O worker pool é a resposta ao critério de custo: em vez de uma goroutine por loja sem limite, N workers fixos consomem o canal jobs — mesmo com 10 mil lojas, CPU, memória e conexões ficam sob controle.

Trade-off honesto: o GC troca controle fino de memória por segurança e produtividade. Em I/O de rede — o domínio deste projeto — isso é irrelevante; em tempo-real duro e embarcado, a previsibilidade do C ainda ganha.

09 Código do projeto

Exemplos — o worker pool

internal/scraper/scraper.go — RunConcurrent
jobs := make(chan model.Store) results := make(chan model.Result) // 1. sobe N workers como goroutines var wg sync.WaitGroup wg.Add(workerCount) for id := 1; id <= workerCount; id++ { go s.worker(ctx, id, jobs, results, &wg) } // 2. produtor enfileira as lojas e fecha o canal jobs go func() { defer close(jobs) for _, store := range stores { jobs <- store } }() // 3. quando os workers terminam, fecha results go func() { wg.Wait(); close(results) }() // 4. coleta na ordem em que chegam — pela VELOCIDADE da loja for r := range results { collected = append(collected, r) }

Quatro papéis — subir workers, enfileirar, fechar, coletar — resolvem uma concorrência controlada que em C exigiria pthreads, mutex e join manuais.

14 Conclusão

Go, em resumo

Go pega a sintaxe familiar do C e troca gerência manual de memória e concorrência frágil por goroutines, channels, worker pool e um GC. Neste projeto I/O-bound, Go e Python empatam (~3.7×) — a vantagem de Go não é velocidade aqui, e sim ter a concorrência como parte central da linguagem: leve, cancelável com context e pronta para escalar em produção.

✓ goroutines ✓ channels + worker pool ✓ context timeout obrigado! 🎯
